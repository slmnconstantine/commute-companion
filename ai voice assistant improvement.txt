================================================================================
  ARCHITECTURE OVERVIEW
================================================================================

CURRENT (problematic):
  Audio → Edge Function (STT + NLU + routing) → Client executes

  Issues:
  - Single opaque call; cannot debug STT/NLU separately
  - Whisper timeout kills entire flow
  - Confirmation loop requires a second full round-trip for yes/no
  - No offline capability

PROPOSED:
  Audio → STT Service (Whisper Large V3 Turbo) → Client-side NLU → Execute

  Benefits:
  - Each stage is independently debuggable and replaceable
  - Faster (no unnecessary Edge Function calls for common commands)
  - Confirmation handled client-side with zero latency
  - Offline fallback for common commands via pattern matching

================================================================================
  PHASE 1 — CORE RELIABILITY (3.5 days)
================================================================================

--------------------------------------------------------------------------
  ITEM 1: Client-Side Intent Matching (2 days) — CRITICAL
--------------------------------------------------------------------------

PRIORITY:   Critical
ESTIMATE:   2 days
IMPACT:     Handles ~80% of commands instantly with zero network latency

DESCRIPTION:
  Move command recognition from the Edge Function to on-device pattern
  matching. Define regex-based intent patterns that cover the most common
  commands. Only fall back to the Edge Function when no pattern matches.

IMPLEMENTATION:
  Define an INTENTS array with pattern-to-command mappings:

    const INTENTS = [
      { pattern: /(?:go|navigate|take me)\s+(?:to\s+)(?:home|my home)/i,
        command: 'NAVIGATE', params: { screen: 'home' } },
      { pattern: /(?:show|find|search)\s+(?:for\s+)?rides/i,
        command: 'SEARCH_RIDES' },
      { pattern: /(?:accept|confirm)\s+(?:the\s+)(?:ride|booking)/i,
        command: 'ACCEPT_BOOKING' },
      { pattern: /(?:post|create|offer)\s+(?:a\s+)?ride/i,
        command: 'PREPARE_RIDE_POST' },
      { pattern: /(?:set|change|update)\s+(?:my\s+)?route/i,
        command: 'SET_ROUTE' },
      // ... additional patterns
    ];

  On receiving a transcript from STT, iterate through INTENTS, check for
  a regex match, and execute the corresponding command locally. Only send
  to the Edge Function if no pattern matches.

BENEFITS:
  - 80% of commands execute instantly with no network latency
  - Fully debuggable — logs show exactly which pattern matched
  - Works offline for all matched commands
  - Eliminates Edge Function calls for the majority of interactions

--------------------------------------------------------------------------
  ITEM 2: Split STT to Dedicated Service (1 day) — CRITICAL
--------------------------------------------------------------------------

PRIORITY:   Critical
ESTIMATE:   1 day
STT MODEL:  Whisper Large V3 Turbo (MIT License)

DESCRIPTION:
  Extract speech-to-text from the Edge Function and route it to a
  dedicated Whisper Large V3 Turbo deployment.

MODEL SPECIFICATIONS:
  - Model:        Whisper Large V3 Turbo
  - License:      MIT (fully open source, commercially permissive)
  - Parameters:   809M
  - Languages:    99+
  - WER:          7.75% (average)
  - Speed:        ~3x faster than Whisper Large V3
  - Deployment:   Self-hosted via faster-whisper or Whisper.cpp

DEPLOYMENT OPTIONS:
  Option A — faster-whisper (Python, CTranslate2 backend):
    - Recommended for server-side deployment
    - Supports batched inference for throughput
    - Can run on CPU or GPU

  Option B — Whisper.cpp:
    - Recommended for edge/on-device inference
    - Pure C/C++ implementation, no Python dependency
    - Runs on Apple Silicon efficiently

  Option C — OpenAI Whisper API (commercial fallback):
    - If self-hosting is not feasible initially
    - Per-minute cost, sends audio to external servers
    - Swap to self-hosted when infrastructure is ready

INTEGRATION:
  - Client sends audio to the STT endpoint (dedicated service)
  - Receives transcript in ~1 second
  - Transcript is then passed to client-side NLU (Item 1)
  - Edge Function only invoked as fallback for complex/unknown queries

BENEFITS:
  - STT failure no longer kills entire NLU/routing pipeline
  - Faster transcription (~1s vs multi-second Edge Function round-trip)
  - Clean separation of concerns

--------------------------------------------------------------------------
  ITEM 3: Button-Based Confirmation Loop (0.5 days) — HIGH
--------------------------------------------------------------------------

PRIORITY:   High
ESTIMATE:   0.5 days
IMPACT:     Eliminates slow re-record loop for confirmations

DESCRIPTION:
  Replace the current confirmation mechanism (which re-records audio and
  sends it to the Edge Function just to detect yes/no) with a dual-input
  approach:

  1. After the assistant speaks, display two large buttons:
       ✓  Confirm
       ✗  Cancel

  2. Simultaneously listen for voice input and perform keyword matching
     on the client side (the transcript is already available).

  3. Resolution logic:
       - User says "yes" or "no" within 5 seconds → match client-side
       - User taps a button → execute immediately
       - Silence for 5 seconds → auto-cancel

BENEFITS:
  - Eliminates an entire Edge Function round-trip
  - Faster, more reliable than re-recording audio
  - Accessible (button fallback for noisy environments)

--------------------------------------------------------------------------
  ITEM 4: Add SET_ROUTE Command (0.5 days) — HIGH
--------------------------------------------------------------------------

PRIORITY:   High
ESTIMATE:   0.5 days
IMPACT:     Spec compliance; hands-free route setting

DESCRIPTION:
  Implement the missing SET_ROUTE command type. Navigate to the
  set-route screen with pre-filled coordinates from voice input.

IMPLEMENTATION:
  Add to command types:

    {
      type: 'SET_ROUTE',
      params: { origin, destination },
      spokenReply: "Setting your route from Session Road to La Trinidad."
    }

  Add corresponding pattern to client-side INTENTS array:

    { pattern: /(?:set|change|update)\s+(?:my\s+)?route/i,
      command: 'SET_ROUTE' }

  When matched, parse origin/destination from the transcript and
  navigate to the set-route screen with pre-filled coordinates.

--------------------------------------------------------------------------
  ITEM 5: Offline Text Fallback (0.5 days) — MEDIUM
--------------------------------------------------------------------------

PRIORITY:   Medium
ESTIMATE:   0.5 days
IMPACT:     Resilience when STT is unavailable

DESCRIPTION:
  The client-side pattern matching (Item 1) already works offline for
  the 5-6 most common commands. The only component requiring a network
  connection is STT (Whisper Large V3 Turbo).

  When STT is unavailable (no network, server down), display a text
  input fallback with the message: "Type your command instead"

  The text input is processed through the same INTENTS pattern matching
  pipeline, so all common commands remain functional offline.
